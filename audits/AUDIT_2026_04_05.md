## 1. Sécurité

### 1.1 Ce qui est solide

#### Authentification & Tokens
- **Mots de passe** : `bcrypt` avec `DEFAULT_COST` (12 rounds) — `auth.rs:49`
- **JWT HS256** : secret en variable d'environnement, durée 7 jours — `auth.rs:127`
- **Validation JWT secret** : `main.rs` vérifie `jwt_secret.len() < 32` au démarrage et appelle `process::exit(1)` si trop court
- **API tokens permanents** : 32 bytes aléatoires, préfixe `nxt_`, stockés en SHA-256, comparaison hachée-à-hachée — `auth.rs:117`
- **Dual-token verify** : `verify_token()` distingue automatiquement JWT de token API — `auth.rs:78`

#### Normalisation & Énumération
- **Email lowercase** : `email.to_lowercase()` avant toute comparaison, bloque le squatting `Alice@` vs `alice@` — `auth.rs:41, 58`
- **Erreurs opaques** : login retourne `"invalid email or password"` qu'il s'agisse d'un email inconnu ou d'un mauvais mot de passe — `auth.rs:64, 71`
- **Register sanitisé** : erreur bcrypt loggée en interne, réponse `"registration failed"` exposée — `auth.rs:52`

#### Signing des binaires
- **Ed25519 (ed25519-dalek v2)** : seed 32 bytes générée aléatoirement, clé publique dérivée à la volée — `signing.rs`
- **Vérification avant extraction** : l'updater vérifie la signature `.sig` avant de décompresser tout binaire téléchargé — `updater.rs`
- **Permissions fichiers 0600** : `~/.nexa/credentials.json` et `~/.nexa/signing_key` — `credentials.rs:66`

#### Registry & Réseau
- **SQL paramétré** : toutes les requêtes `sqlx` utilisent `$1`, `$2`, etc. — aucune interpolation de chaîne — `postgres.rs`
- **Rate limiting** : `tower_governor` — burst=3, 1 req/6s par IP sur `/auth/register` et `/auth/login` — `http.rs:63-64`
- **Validation nom de package** : regex `^[a-zA-Z0-9][a-zA-Z0-9._-]{0,213}$`, bloque `..` et traversée de répertoire — `packages.rs`
- **Limite taille bundle** : 50 MB max, rejet HTTP 413 au-delà — `packages.rs`
- **Validation email** : `valid_email()` vérifie `@`, parties non-vides, domaine avec `.`, max 254 chars (RFC 5321) — `auth.rs`

#### Transport
- `reqwest` configuré avec feature `rustls-tls` — pas d'OpenSSL, surface d'attaque réduite
- Images Docker en user non-root

## 2. Infrastructure & CI/CD

### 2.1 Workflows GitHub Actions

Trois workflows couvrent l'intégralité du cycle de vie :

#### `snapshot.yml` — déclencheur : push main, PR, dispatch manuel
```
Jobs : fmt → clippy → test → wasm-validate (parallèles)
         ↓ (si pas PR)
       docker → build (5 plateformes) → snapshot-release
```
- `cargo fmt --all -- --check` bloquant
- `cargo clippy --all -- -D warnings` bloquant
- `cargo test --all` bloquant
- `wasm-validate` : installe `wabt` + `wasmtime`, exécute `cargo test -p nexa-compiler validate_wasm_binary` — **bloquant**
- `coverage` : `cargo-tarpaulin` + upload Codecov — `continue-on-error: true` (non bloquant, intentionnel)

#### `release.yml` — déclencheur : dispatch manuel avec input version
- Builds reproductibles `--locked` sur 5 cibles :
  - `x86_64-unknown-linux-musl`
  - `aarch64-unknown-linux-musl`
  - `x86_64-apple-darwin`
  - `aarch64-apple-darwin`
  - `x86_64-pc-windows-msvc`
- SHA-256 checksums générés pour chaque archive
- SLSA provenance attestation (`id-token: write`)

#### `deploy-registry.yml` — déclencheur : changements dans `crates/registry/`
- Build image Docker → push `ghcr.io`
- Déploiement SSH sur VPS → `docker-compose up`

### 2.2 Points forts de l'infra

- **Builds reproductibles** : `cargo build --release --locked` — le `Cargo.lock` est authoritative
- **Cache Cargo** : `actions/cache@v4` par job, keyed sur `Cargo.lock`
- **Multi-arch** : binaires ARM64 natifs (ubuntu-24.04-arm)
- **SBOM + provenance** : `sbom: true`, `provenance: true` dans docker build
- **Non-root Docker** : user `cargo` dans `Dockerfile.cli`
- **Strip binaires** : `strip "$BIN" 2>/dev/null || true` sur les builds Unix

## 3. Architecture

### 3.1 Structure du workspace

```
Cargo workspace (4 crates)
├── crates/compiler/   — Pipeline compilateur (7 078 LOC)
├── crates/cli/        — Interface CLI (nexa build/run/publish/…)
├── crates/server/     — Serveur de développement (Axum, HMR)
└── crates/registry/   — Registry de packages (Axum + PostgreSQL)
```

Chaque crate suit **Clean Architecture** avec trois couches :
```
domain/          → Entités (AST, IR, Span) — aucune dépendance externe
application/     → Services + Ports (traits abstraits)
infrastructure/  → Implémentations concrètes (FS, HTTP, DB)
```

### 3.2 Pipeline compilateur (8 étapes)

```
Source .nx
  ↓  Lexer        (lexer.rs     ~694 LOC)  → flux de tokens typés
  ↓  Parser       (parser.rs   ~1289 LOC)  → AST typé (89 nœuds)
  ↓  Resolver     (resolver.rs  ~501 LOC)  → résolution multi-fichiers + dédoublonnage
  ↓  SemanticAnalyzer (semantic.rs ~1905 LOC)
         Pass 1 : collecte des noms (classes, interfaces)
         Pass 2 : validation extends/implements
         Pass 3 : validation imports/routes
         Pass 4 : vérification visibilité (public/private)
         Pass 5 : vérification annotations de type + types retour
         Pass 6 : validation paramètres génériques
         Pass 7 : inférence Hindley-Milner (Damas-Milner Algorithm W)
  ↓  Optimizer    (optimizer.rs  ~664 LOC)  → 4 passes
         Pass 1 : suppression code mort (analyse de portée depuis les routes)
         Pass 2 : inlining de composants
         Pass 3 : aplatissement d'arbre
         Pass 4 : précalcul de props (constant folding)
  ↓  Lowering     (lower.rs)               → AST → IrModule
  ↓  Codegen      (2 backends)
         JS   : codegen.rs   (~380 LOC) + runtime embarqué (365 LOC)
         WASM : wasm_codegen/ (~1164 LOC) + GC Cheney semi-espace
```

### 3.3 Inférence de types — Hindley-Milner complet

Implementation de **Damas-Milner Algorithm W** avec :
- `HmType` — représentation interne avec variables d'unification `Var(u32)`
- `TypeScheme` — schéma universel `∀α₁…αₙ. τ`
- `HmEnv` — environnement `String → TypeScheme`
- `Unifier` — map de substitution avec `unify()` + **occurs-check** (protection contre types infinis)
- `generalize()` — fermeture universelle des variables libres hors env (règle `let`)
- `instantiate()` — remplacement des variables ∀ par des variables fraîches (règle `var`)
- `SemanticError::InfiniteType` — erreur levée si occurs-check échoue

### 3.4 Backend WASM — GC générationnel Cheney

Layout mémoire (15 pages × 64 KiB = 960 KiB) :
```
[string_pool]           → pool de chaînes littérales (section data)
[nursery FROM-space]    → 128 KiB — allocation courante
[nursery TO-space]      → 128 KiB — destination du GC mineur
[old generation]        → 512 KiB
[shadow stack]          → 64 KiB — racines GC sur la pile d'appels WASM
[remembered set]        → 64 KiB — pointeurs old-gen → nursery
```

Algorithme :
1. Sauvegarder `from_ptr` comme frontière
2. Réinitialiser `to_ptr = to_start`
3. Tracer les racines : shadow stack + remembered set
4. Scanner `to-space[to_start, to_ptr)` avec `$gc_scan_object`
5. Échanger from/to semi-espaces

### 3.5 Format NXB (Nexa Binary)

```
[b"NXB"] [version: u8] [bincode v2 payload (Program sérialisé)]
```
- Magic header 3 bytes + byte de version pour compatibilité ascendante
- `bincode::serde` v2 — jamais de panic sur données corrompues
- `PackageError::FormatVersion` avec message explicite si version inconnue

### 3.6 Resolver multi-fichiers

- Mapping `com.myapp.models.User` → chemin fichier
- Cache `HashMap<PathBuf, Vec<Declaration>>` — évite les re-parsings
- Cycle detection via `HashSet<PathBuf>` (loading stack) → `ResolveError::Cycle`
- Stratégie de recherche : `src/main/`, `lib/`, `modules/<name>/lib/`
- Trait `SourceProvider` — abstraction FS (prod) vs in-memory (tests)

### 4.2 Points résolus le 2026-04-07

#### ~~Clones dans le vérificateur de types~~ — ✅ RÉSOLU

`semantic.rs` — commentaires de justification ajoutés dans `check_stmt_types` et `check_stmt_hm` aux quatre emplacements (`if`/`while`/`for`). Le compromis est maintenant documenté explicitement : scopes de 5–20 bindings, une fois par structure de contrôle par méthode, piste d'optimisation future (`Cow<HashMap>`) mentionnée in situ.

#### ~~Substitution HM sans path compression~~ — ✅ RÉSOLU

`semantic.rs` — `Unifier::apply` passe de `&self` à `&mut self` avec **path compression** :
```rust
// Avant : O(n) à chaque appel sur une chaîne α1→α2→α3→Int
// Après  : α1 pointé directement sur Int après le premier suivi — O(1) amorti
Some(bound) => {
    let resolved = self.apply(&bound);
    self.subst.insert(id, resolved.clone()); // path compress
    resolved
}
```
`generalize()` mis à jour en `u: &mut Unifier`. 254 tests : 0 régressions.

#### ~~Span dummy dans TypeMismatch~~ — ✅ RÉSOLU

`ast.rs` — `Stmt::Return(Option<Expr>)` refactorisé en :
```rust
Stmt::Return {
    expr: Option<Expr>,
    #[serde(skip)]   // pas de bump de version NXB
    span: Span,
}
```
Le parser capture la span du mot-clé `return` et la propage. `check_stmt_types` et `check_stmt_hm` utilisent `*span` au lieu de `Span::dummy()`. `Span` obtient `serde::{Serialize, Deserialize}`. Optimizer, lower, et tous les tests mis à jour. 254 tests : 0 régressions.

#### ~~`Option<Int>` dans stdlib — typage monomorphe~~ — ✅ RÉSOLU

`stdlib/src/option.nx` est maintenant `class Option<T>` avec `private T value;`. Deux correctifs compilateur nécessaires :
- **`lower.rs`** : `x = expr` émettait `x.x = expr` en JS — corrigé par guard `Expr::Ident(n) if n == field`
- **`parser.rs`** : `private T value;` non reconnu — corrigé par `is_custom_type_field_start()` (lookahead 2 tokens)
- **`parser.rs`** : `this.data.push(item)` ne parsait pas comme statement — corrigé en unifiant le cas `Token::This` dans la branche générique `_` qui appelle `parse_expr()`

`Option<T>`, `Result<T>`, `Stack<T>`, `Queue<T>` sont tous génériques. 255 tests : 0 régressions.

### 6.1 Compilateur

#### ~~A — Compléter la stdlib (impact : Haut)~~ — ✅ FAIT le 2026-04-05
`io`, `math`, `str`, `option<T>`, `result<T>` complètement réécrits. `collections.nx` ajouté (`Stack<T>` + `Queue<T>`). `Print.log()` compile vers `log()` du runtime JS. Deux bugs compilateur corrigés : réassignation de variable locale (`lower.rs`) et champs de type personnalisé (`parser.rs`). 255 tests : 0 régressions.

#### B — Pattern Matching + ADTs (impact : Haut)
Ajouter :
```nexa
enum Color { Red, Green, Blue }
match expr {
    Color.Red   => { ... }
    Color.Green => { ... }
    _           => { ... }
}
```
- Prérequis côté AST : `EnumDecl`, `MatchExpr`, `MatchArm`
- Semantic : exhaustiveness checking
- Codegen JS : `switch` ou conditions chaînées
- Impact positif sur `Option`/`Result` de la stdlib

#### C — Récupération d'erreurs dans le Parser (impact : Moyen)
Le parser s'arrête à la première erreur syntaxique. Une stratégie de synchronisation sur des délimiteurs (`}`, `;`) permettrait de rapporter plusieurs erreurs en une passe.
- Pattern classique : _panic mode recovery_ sur les tokens de synchronisation

#### ~~D — Spans précises dans TypeMismatch~~ — ✅ FAIT le 2026-04-07
`Stmt::Return` porte maintenant sa span (`#[serde(skip)]`). Le parser capture la span du mot-clé `return`. `check_stmt_types` et `check_stmt_hm` propagent `*span` dans `TypeMismatch`. Plus de `Span::dummy()` pour les erreurs de retour.

#### ~~E — Path compression dans l'unificateur HM~~ — ✅ FAIT le 2026-04-07
`Unifier::apply` implémente la path compression (style union-find). Complexité : O(1) amorti au lieu de O(n) par appel.

#### F — `nexa test` — runner de tests intégré (impact : Haut)
Ajouter une syntaxe de test dans le langage :
```nexa
test "addition works" {
    let result = 1 + 1;
    assert(result == 2);
}
```
Et la commande `nexa test` pour les exécuter. Sans cela, les utilisateurs ne peuvent pas tester leur code Nexa nativement.

#### G — `nexa fmt` — formateur de code (impact : Moyen)
Un formateur officiel garantit une base de code cohérente. Peut s'appuyer sur le parser existant + émission AST → source formatée.

### 6.3 LSP & Tooling

#### L — Language Server Protocol (impact : Très haut, effort : Haut)
